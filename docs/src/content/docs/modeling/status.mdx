Every node in your model can have a status that tracks where it is in the implementation lifecycle.

## The three statuses

<span class="status-pill status-proposed">Proposed</span> <span class="status-pill status-wip">WIP</span> <span class="status-pill status-ready">Ready</span>

### Proposed

<span class="status-pill status-proposed">Proposed</span> — this node is planned but doesn't exist in code yet. It's a design decision waiting to be built.

### WIP (Work in Progress)

<span class="status-pill status-wip">WIP</span> — code exists for this node, but it might be incomplete. Stubs, partial implementations, scaffolding — anything that's started but not finished.

When an AI agent implements a node, it marks it as WIP with a **reason** explaining what was built: "Basic CRUD endpoints, needs auth middleware and rate limiting."

### Ready

<span class="status-pill status-ready">Ready</span> — this node is production-ready. The implementation is complete, tests pass, and all contract items are satisfied.

**Ready is gated.** A node can only be marked ready when every inherited **expect** contract item has `passed: true`. If even one expect item is unchecked or failed, the ready status is blocked.

## What gets status

- Systems, containers, components, operations, processes, models
- Persons and external systems don't get status (they're outside your control)

Marking a system or container as external automatically clears its status.

## Nodes without status

Nodes without any status appear gray and unactionable. Use "no status" for framework defaults or infrastructure that doesn't need implementation effort — things that are just *there*.

In the sidebar tree, the same colored dots appear next to node names for a quick overview of project progress.

## Status in the AI workflow

During the [implementation workflow](/scryer/ai/implementation-workflow/):

1. All new nodes start as **proposed**
2. The agent builds each piece and marks it **WIP** with a reason
3. When you're satisfied, ask the agent to **verify** — it checks for stubs, runs tests, and confirms contract items
4. If everything checks out, the node moves to **ready**

Status propagates upward: when all components in a container are WIP or ready, the task system prompts the agent to mark the container as WIP too.
